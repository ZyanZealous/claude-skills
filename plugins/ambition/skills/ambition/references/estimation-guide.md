# 估点与容量规划指南

BREAKDOWN 阶段用。T恤尺码映射故事点：XS(1) / S(2) / M(3) / L(5) / XL(8)。

## 校准基准例子

估点的本质是**相对比较**，不是预测小时数。用下面的锚定例子校准（按全栈 Web 项目调校，其他领域类比迁移）：

### XS = 1pt（< 半天，几乎无未知数）
- 加一个配置项 / 环境变量
- 改文案、调样式、换图标
- 给已有列表加一个排序选项
- 写一个简单的数据库迁移（加列）

### S = 2pt（半天–1天，方案明确）
- 单表 CRUD API（已有同类参考可抄）
- 一个无复杂交互的静态页面
- 给已有 API 加分页/筛选参数
- 接入一个文档清晰的第三方 SDK（如对象存储上传）

### M = 3pt（1–2 天，有少量设计决策）
- 完整的注册/登录流程（API + 页面 + 校验）
- 一个带表单校验、多状态的交互页面
- 设计并实现一组关联表（3–4 张）及其 API
- 文件上传 + 处理（缩略图/格式校验）

### L = 5pt（2–3 天，有明显未知数或跨多模块）
- 第三方支付接入（含回调、对账、异常处理）
- 实时功能（WebSocket 通知/协作）
- 复杂业务流程（多状态审批流）
- 性能优化任务（先要定位，方案不确定）

### XL = 8pt（> 3 天）→ **必须继续拆**
XL 不是合法的最终估点，是"还没拆完"的标记。拆法：
- 按层拆：数据层 / API / 前端 / 集成
- 按流程拆：happy path 一个任务，每类异常处理单独任务
- 按里程碑拆：先做 demo 版（写死数据），再做真实版

## 估点修正系数

在基准上有这些情况就**上调一档**：
- 涉及没用过的技术/库（学习成本）
- 依赖外部团队或第三方审批（等待 + 沟通成本）
- 需求里有"大概""可能""到时候再说"（隐藏的返工）
- 触碰核心数据（要更谨慎的迁移和回滚方案）

这些情况**不要下调**估点（常见自欺）：
- "我以前做过类似的"——环境不同，集成成本还在
- "AI 可以帮我写"——生成快但调试验证时间还在

## Sprint 容量规则

### 基础容量
- 以 **2 周 sprint** 为默认。1 人全职 ≈ 每 sprint 完成 **15–20pt**（含沟通、测试、修 bug 的真实产出，不是理想编码时间）
- 第一个 sprint **按 60% 容量装载**（10–12pt）：没有历史速度数据，宁可提前完成追加任务，不要首战延期
- 兼职投入按比例折：每周 2 个晚上 ≈ 0.2 人 ≈ 3–4pt/sprint

### 速度校准（从 Sprint 2 开始）
```
速度 = 最近 2–3 个 sprint 实际完成点数的平均值
下个 sprint 装载量 = 速度 × 0.9（留 10% 缓冲）
```
- 完成率连续 > 110% → 估点偏保守，整体下调一档重新校准
- 完成率连续 < 70% → 不要先怪执行，先查：估点漏了修正系数？阻塞没及时处理？还是任务拆得太大？

### 装载规则
- **Must 故事优先填满**，Should 只在 Must 装完且还有容量时进入
- 每个 sprint 至少包含一个**端到端可演示**的成果（不要一个 sprint 全是底层任务，看不到任何用户可见的进展）
- 有依赖链的任务（T001→T002→T003）不要全塞进同一个 sprint 还指望并行
- 留 1–2pt 给"sprint 中冒出来的小事"——它们一定会出现

### 反模式警告
- **点数通胀**：为了"显得有产出"把 XS 估成 M。点数只用于规划，不是 KPI，通胀只会毁掉速度校准。
- **100% 装载**：满载 = 任何意外都导致延期。计划本身要有弹性。
- **跨 sprint 的半成品**：一个任务连续两个 sprint 都是 IN_PROGRESS，说明它实际是 XL，停下来拆掉它。

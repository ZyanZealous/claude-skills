# DISCOVER 问题库

六类问题，按深度选取。原则：先从用户描述提取已知答案，只问真正不知道的；一次最多 5 个问题，分批进行。

## 按深度选取

| 深度 | 问法 |
|------|------|
| quick | 每类挑 ★ 标记的核心问题，总共 5–8 个，一轮问完 |
| standard | ★ + 大部分常规问题，2–3 轮 |
| full / enterprise | 全部维度，可多轮；enterprise 额外做干系人清单 |

---

## 1. 问题域（Problem）

- ★ 这个产品解决什么问题？现在的人们怎么解决（不用你的产品时）？
- ★ 如果这个问题不解决，最坏会怎样？（衡量痛点强度）
- 这个问题发生的频率？（每天 / 偶尔 / 一年一次）
- 已经有竞品/替代品吗？它们差在哪？
- 为什么是现在做？（什么变化让这个时机成立）

## 2. 用户（Users）

- ★ 主要用户是谁？（具体到角色，"运营人员"比"企业用户"好）
- ★ 有几类不同的用户？他们的诉求冲突吗？（如买家 vs 卖家）
- 用户的技术水平？（决定 UI 复杂度容忍度）
- 用户在什么设备/环境下使用？（手机/PC/平板，办公室/户外）
- 谁付钱？付钱的人和使用的人是同一个吗？
- 预期用户量级？（10 人内部工具和 10 万 C 端产品是两个物种）

## 3. 功能（Features）

- ★ 如果只能保留一个功能，是哪个？（识别真正的核心）
- ★ 第一个版本上线的最低标准是什么？
- 有没有"看起来必须但其实可以砍"的功能？（主动帮用户砍）
- 需要哪些和外部系统的集成？（支付/登录/通知/已有内部系统）
- 内容/数据从哪来？（用户生成 / 运营录入 / 第三方接口 / 爬取）
- 需要后台管理界面吗？谁用？

## 4. 非功能性（NFR）

- ★ 多少人同时用？高峰是什么时候？
- 数据丢了多大事？（决定备份和事务严格度）
- 响应速度有体感要求吗？（实时协作 vs 报表系统差别巨大）
- 需要支持哪些浏览器/系统版本？
- 有合规要求吗？（GDPR / 等保 / 行业规范 / 数据出境）
- 无障碍要求？（政府/教育项目常有硬性要求）

## 5. 技术（Technical）

- ★ 技术栈有偏好或限制吗？（团队已有技能 / 公司规定）
- 是全新系统还是接入已有系统？已有系统什么技术？
- 部署到哪？（自有服务器 / 云 / 用户本地）
- 有已经定了的技术决策吗？（"必须用公司的 SSO"这类）
- 团队几个人，分别擅长什么？
- 预算限制？（影响云服务和第三方付费 API 的选择）

## 6. 风险与约束（Risks & Constraints）

- ★ 死线是什么？是硬死线（发布会/合同）还是软目标？
- ★ 这个项目最可能死在哪？（让用户自己说出最大的担忧）
- 有需要提前申请/审批的资源吗？（API 配额、域名备案、应用商店审核）
- 哪些决策不归你定？决策人是谁、响应快吗？
- 项目中途最可能变的是什么？（提前为变化设计弹性）

## 提问技巧

- **给选项而不是开放题**：问"预计日活是百级、千级还是万级以上？"比"预计多少用户？"更容易回答。
- **用假设换确认**："我假设第一版不需要多语言，对吗？"比"需要多语言吗？"推进更快——这也直接生成 assumption log 条目。
- **追问一层**：用户说"要快"，追问"快是指页面秒开，还是指功能两周内上线？"
- **答不上来的不死磕**：标记为开放问题或假设，继续走。DISCOVER 的目标是 80% 清晰度，不是 100%。

## 产出检查

`discovery.md` 完成时应包含：
- [ ] 每类问题至少覆盖了 ★ 项（或显式记录"不适用"）
- [ ] assumption log 里没有"隐形假设"——你做的每个判断都写出来了
- [ ] 开放问题都有负责人或"带假设推进"的标记
- [ ] 向用户做了一次总结复述："我理解的需求是……有偏差吗？"
